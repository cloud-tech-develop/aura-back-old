## 👤 Historia de Usuario

**Como** sistema de punto de venta  
**Quiero** generar automáticamente una factura electrónica al completarse una venta  
**Para** garantizar el cumplimiento tributario, mantener trazabilidad de las transacciones y simplificar el proceso de facturación para el usuario

---

## 🧠 Descripción Funcional

Al confirmarse una venta en el sistema Aura POS, el backend debe crear automáticamente un registro de factura electrónica con todos los datos relevantes de la transacción. La factura debe incluir: prefijo y consecutivo únicos, método de pago, valores calculados (subtotal, descuento, impuesto, total), datos del usuario que registra, empresa y sucursal, fecha y hora de emisión, y estado inicial Pendiente de validación DIAN.

El sistema debe generar el CUFE (Código Único de Factura Electrónica) utilizando el algoritmo SHA-384 que combina datos de la factura. Además, debe registrar automáticamente los pagos asociados a la venta en la tabla de ReciboPago (factura_pagos), creando la relación entre la factura y sus pagos.

La factura debe persistirse con soft delete para mantener auditoría, y debe soportar los ambientes de desarrollo y producción para la integración futura con la DIAN.

---

## ✅ Criterios de Aceptación

### Escenario 1: Generación exitosa de factura al completar venta
- Dado que existe una venta completada con detalles, método de pago y valor
- Cuando el sistema procesa la confirmación de la venta
- Entonces se debe crear automáticamente un registro en la tabla `factura` con:
  - Prefijo correspondiente a la sucursal/empresa
  - Consecutivo secuencial y único por prefijo
  - Método de pago extraído de la venta
  - Valor total calculado (suma de detalles menos descuentos)
  - Descuento total aplicado
  - usuario_id del vendedor que registró la venta
  - empresa_id extraído del JWT
  - cufe generado con algoritmo SHA-384
  - fecha_hora_emision con timestamp actual
  - estado_dian como PENDIENTE
  - tipo_ambiente según configuración (dev/prod)
  - created_at con timestamp actual

### Escenario 2: Generación de CUFE válido
- Dado que se procesa una factura con todos sus datos
- Cuando se genera el código CUFE
- Entonces el CUFE debe contener:
  - Número de factura (prefijo + consecutivo)
  - Fecha y hora de emisión
  - Nit del emisor (empresa)
  - Nit del receptor (cliente si existe)
  - Valor total de la factura
  - Código SHA-384 resultante

### Escenario 3: Registro automático de pagos
- Dado que la venta incluye pagos registrados
- Cuando se genera la factura
- Entonces debe crearse un registro en `factura_pago` (ReciboPago) por cada pago con:
  - factura_id referenciando la factura creada
  - valor del pago
  - banco (si aplica para transferencia/tarjeta)
  - tipo (credito/debito para tarjetas)
  - metodo_pago del pago
  - descripcion si existe
  - usuario_id del registrador
  - created_at con timestamp

### Escenario 4: Manejo de métodos de pago múltiples
- Dado que una venta tiene múltiples pagos (ej: efectivo + tarjeta)
- Cuando se genera la factura
- Entonces debe crear un ReciboPago por cada método de pago
- Y el total de pagos debe ser igual al valor total de la factura

### Escenario 5: Trazabilidad y auditoría
- Dado que se genera una factura
- Cuando se consulta posteriormente
- Entonces debe mantener:
  - created_at con fecha de creación
  - updated_at con última modificación
  - deleted_at null (no eliminada)
- Y si se elimina (soft delete):
  - deleted_at con fecha de eliminación
  - Mantener datos para auditoría

### Escenario 6: Consecutivo único por prefijo
- Dado que existen múltiples facturas para una empresa
- Cuando se genera una nueva factura con un prefijo específico
- Entonces el consecutivo debe ser secuencial y no repetirse
- Y debe validarse que no haya conflictos de concurrencia

---

## ❌ Casos de Error

- Si la venta no tiene detalles/items, no debe generarse factura y debe lanzarse excepción
- Si el prefijo de facturación no existe para la sucursal, debe retornarse error 400
- Si el consecutivo supera el límite permitido (ej: 999999999), debe marcarse como error
- Si falla la generación del CUFE, debe registrar el error y marcar factura como ERROR
- Si la empresa no tiene configuración de facturación electrónica, no debe permitir ventas con factura
- Si el JWT no contiene empresa_id válido, debe rechazarse la operación con 401

---

## 🔐 Reglas de Negocio

- El prefijo de facturación se obtiene de la configuración de sucursal
- El CUFE se genera con algoritmo SHA-384 usando: NumFactura + FechaEmision + NitEmisor + NitReceptor + ValorTotal + Timestamp + ClaveTech
- El estado_dian inicial es siempre PENDIENTE
- El tipo_ambiente se determina por configuración de empresa (dev por defecto)
- Los pagos se vinculan a la factura mediante factura_id (relación 1:N)
- Soft delete obligatorio: nunca eliminar físicamente un registro de factura
- El descuento se aplica a nivel de factura (no por item)

## 🗄️ Schema de Base de Datos (PostgreSQL)

### Tabla: factura

```sql
CREATE TABLE factura (
    id BIGSERIAL PRIMARY KEY,
    prefijo VARCHAR(10) NOT NULL,
    consecutivo BIGINT NOT NULL,
    metodo_pago VARCHAR(50) NOT NULL CHECK (metodo_pago IN ('efectivo', 'transferencia', 'tarjeta', 'credito')),
    valor DECIMAL(12,2) NOT NULL,
    descuento DECIMAL(12,2) DEFAULT 0,
    usuario_id INTEGER NOT NULL REFERENCES usuario(id),
    empresa_id INTEGER NOT NULL REFERENCES empresa(id),
    venta_id BIGINT NOT NULL REFERENCES venta(id),
    cufe VARCHAR(100) UNIQUE NOT NULL,
    descripcion VARCHAR(300),
    fecha_hora_emision TIMESTAMP NOT NULL DEFAULT NOW(),
    estado_dian VARCHAR(50) NOT NULL DEFAULT 'PENDIENTE' CHECK (estado_dian IN ('PENDIENTE', 'AUTORIZADO', 'RECHAZADO', 'ERROR')),
    tipo_ambiente VARCHAR(10) NOT NULL DEFAULT 'dev' CHECK (tipo_ambiente IN ('dev', 'prod')),
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP,
    deleted_at TIMESTAMP,
    
    CONSTRAINT factura_empresa_fk FOREIGN KEY (empresa_id) REFERENCES empresa(id),
    CONSTRAINT factura_usuario_fk FOREIGN KEY (usuario_id) REFERENCES usuario(id),
    CONSTRAINT factura_venta_fk FOREIGN KEY (venta_id) REFERENCES venta(id),
    CONSTRAINT factura_cufe_unique UNIQUE (cufe),
    CONSTRAINT factura_prefijo_consecutivo_unique UNIQUE (empresa_id, prefijo, consecutivo)
);

CREATE INDEX idx_factura_empresa ON factura(empresa_id);
CREATE INDEX idx_factura_venta ON factura(venta_id);
CREATE INDEX idx_factura_cufe ON factura(cufe);
CREATE INDEX idx_factura_deleted_at ON factura(deleted_at) WHERE deleted_at IS NULL;
```

### Tabla: factura_pago (ReciboPago)

```sql
CREATE TABLE factura_pago (
    id BIGSERIAL PRIMARY KEY,
    factura_id BIGINT NOT NULL REFERENCES factura(id),
    valor DECIMAL(12,2) NOT NULL,
    banco VARCHAR(300),
    tipo VARCHAR(50) CHECK (tipo IN ('credito', 'debito')),
    descripcion VARCHAR(300),
    usuario_id INTEGER NOT NULL REFERENCES usuario(id),
    metodo_pago VARCHAR(50) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP,
    
    CONSTRAINT factura_pago_factura_fk FOREIGN KEY (factura_id) REFERENCES factura(id) ON DELETE CASCADE
);

CREATE INDEX idx_factura_pago_factura ON factura_pago(factura_id);
```
