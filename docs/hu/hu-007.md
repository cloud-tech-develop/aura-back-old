## 👤 Historia de Usuario

**Como** sistema de facturación  
**Quiero** registrar logs de todos los estados y eventos de las facturas sin afectar el rendimiento  
**Para** mantener trazabilidad completa del ciclo de vida de la factura, auditorías y debugging sin bloquear el proceso principal

---

## 🧠 Descripción Funcional

El sistema debe implementar un mecanismo de logging asíncrono para registrar todos los eventos relacionados con las facturas. Este sistema debe ser completamente independiente del flujo principal de la aplicación, no debe agregar latencia al proceso de creación de facturas, y debe permitir consultar el historial de cambios de cada factura.

Los logs deben registrar: estado actual, estado anterior (si aplica), evento/acción que genera el log, datos de la factura en el momento del evento, usuario que ejecutó la acción, timestamp, y metadata adicional relevante.

---

## ✅ Criterios de Aceptación

### Escenario 1: Logging asíncrono de creación de factura
- Dado que se procesa una venta que genera una factura
- Cuando se crea la factura exitosamente
- Entonces:
  - El sistema debe registrar el evento en background sin bloquear la respuesta
  - El log debe incluir: factura_id, evento "CREACION", estado "PENDIENTE", datos completos de la factura, timestamp
  - El tiempo de respuesta del endpoint no debe incrementarse

### Escenario 2: Log de cambio de estado DIAN
- Dado que existe una factura con estado "PENDIENTE"
- Cuando se actualiza el estado a "AUTORIZADO" o "RECHAZADO"
- Entonces:
  - Debe crearse un registro de log con:
    - evento "CAMBIO_ESTADO_DIAN"
    - estado_anterior: "PENDIENTE"
    - estado_nuevo: "AUTORIZADO" o "RECHAZADO"
    - metadata adicional (respuesta de DIAN si aplica)

### Escenario 3: Log de envío a DIAN
- Dado que se intenta enviar una factura a la DIAN
- Cuando se procesa el envío
-Entonces:
  - Debe crearse un log con evento "ENVIO_DIAN"
  - Debe incluir: intentos, respuesta, tiempo de respuesta, éxito/fallo

### Escenario 4: Consulta de historial de logs
- Dado que existe una factura con múltiples logs
- Cuando se consultan los logs de esa factura
-Entonces:
  - Se deben retornar todos los logs ordenados por fecha descendente
  - El sistema debe permitir filtrar por tipo de evento

### Escenario 5: Aislamiento de fallos
- Dado que el servicio de logging falla
- Cuando se intenta registrar un log
- Entonces:
  - El proceso principal NO debe fallar
  - El error de logging debe ser capturado silenciosamente
  - No debe afectar la experiencia del usuario

---

## ❌ Casos de Error

- Si la factura no existe, debe retornar 404
- El logging NUNCA debe lanzar excepción al proceso principal
- Si la cola de mensajes está llena, debe hacer fallback a logging síncrono

## 🗄️ Schema de Base de Datos (PostgreSQL)

### Tabla: factura_log

```sql
CREATE TABLE factura_log (
    id BIGSERIAL PRIMARY KEY,
    factura_id BIGINT NOT NULL REFERENCES factura(id),
    evento VARCHAR(50) NOT NULL,
    estado_anterior VARCHAR(50),
    estado_nuevo VARCHAR(50),
    datos JSONB,
    usuario_id INTEGER,
    mensaje VARCHAR(500),
    metadata JSONB,
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_factura_log_factura ON factura_log(factura_id);
CREATE INDEX idx_factura_log_evento ON factura_log(evento);
CREATE INDEX idx_factura_log_created_at ON factura_log(created_at);

-- Partición opcional por fecha para mejor rendimiento
-- CREATE INDEX idx_factura_log_factura_fecha ON factura_log(factura_id, created_at DESC);
```

```java
            String estadoNuevo, Object datos, Integer usuarioId, String mensaje) {
        // Lógica de guardado
    }
}
```

### Configuración de Executor

```java
@Configuration
@EnableAsync
public class AsyncConfig {
    
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("factura-log-");
        executor.initialize();
        return executor;
    }
}
```

### Characteristics

- **No bloqueante**: Usa `@Async` para ejecución en background
- **Tolerante a fallos**: Try-catch con logging de errores internos
- **Rendimiento**: Pool de hilos dedicado para no competir con otros procesos
- **Ordenado**: Los logs se insertan en orden por timestamp
