## 👤 Historia de Usuario

**Como** usuario del sistema  
**Quiero** asignar locales a vendedores (vendedor actual y vendedor anterior)  
**Para** mantener el historial de asignación y facilitar la gestión de rutas

---

## 🧠 Descripción Funcional

El sistema debe permitir asociar un local a un vendedor actual y mantener registro del vendedor anterior. Esto permite mantener un historial de asignación y facilita la transición cuando un vendedor deja de atender un local.

---

## ✅ Criterios de Aceptación

### Escenario 1: Asignar local a vendedor
- Dado que el usuario está en el formulario de crear/editar local
- Cuando selecciona un vendedor del dropdown de "Vendedor Actual"
-Entonces el sistema asocia el local al vendedor seleccionado

### Escenario 2: Reasignar local a nuevo vendedor
- Dado que un local tiene un vendedor actual asignado
- Cuando el usuario selecciona un nuevo vendedor
-Entonces el sistema mueve el vendedor anterior a "Vendedor Anterior" y asigna el nuevo como "Vendedor Actual"

### Escenario 3: Ver historial de vendedores
- Dado que el usuario está viendo los detalles de un local
- Cuando visualiza la información del local
-Entonces el sistema muestra el vendedor actual y el vendedor anterior (si existe)

### Escenario 4: Listar vendedores en dropdown
- Dado que el usuario está en el formulario de local
- Cuando carga el dropdown de vendedores
-Entonces el sistema muestra solo los empleados con tipo "Vendedor" activos

---

## ❌ Casos de Error

- Si no hay vendedores disponibles, mostrar mensaje "No hay vendedores configurados"
- Si el vendedor seleccionado no existe, mostrar error

---

## 🔐 Reglas de Negocio

- El vendedor anterior se guarda automáticamente al hacer una nueva asignación
- Solo se muestran empleados con tipo "Vendedor" en los dropdowns
- Se filtran por empresa_id del JWT

---

## 🎨 Consideraciones UI/UX

- Mostrar el nombre del vendedor anterior en la vista de detalles
- Permitir visualizar el historial de cambios
- Desplegar tooltip con información del vendedor anterior

---

## 📡 Requisitos Técnicos

- El endpoint POST/PUT /api/locales debe aceptar vendedor_actual_id y vendedor_anterior_id
- El endpoint GET /api/locales/{id} debe devolver ambos campos de vendedor
