# HU-014 - Sistema de Precios Dinámicos por Cliente

## Contexto

Actualmente el sistema maneja precios estáticos a través de listas de precios (EP-001 - HU-001), donde cada producto tiene un precio definido para una lista específica. Sin embargo, no existe la capacidad de asignar precios especiales a clientes individuales o por segmentos de clientes.

Esta HU permite implementar un sistema de precios dinámicos que permita:
- Precios especiales para clientes recurrentes (Precio 1)
- Descuentos específicos por cliente (Precio 2)  
- Precios por volumen (Precio 3)

## Epic Relacionado

EP-004 - Módulo de Cuentas por Cobrar y Pagar

## User Story

### Descripción

Como administrador del sistema, necesito poder configurar precios especiales para clientes individuales o por segmentos, de manera que se apliquen automáticamente al momento de realizar ventas, para ofrecer precios diferenciados según el tipo de cliente y sus condiciones comerciales.


### 1. Configuración de Precios Especiales
- **CA-1.1**: Debe existir un endpoint para crear/editar precios especiales por cliente
- **CA-1.2**: Debe poder configurar múltiples tipos de precios para un mismo cliente
- **CA-1.3**: Debe poder configurar precios por producto individual o por categoría
- **CA-1.4**: Debe poder configurar precios por volumen con rangos de cantidad
- **CA-1.5**: Debe poder activar/desactivar reglas de precios sin eliminarlas

#### 2. Tipos de Precios Especiales
- **CA-2.1**: Precio 1 - Cliente recurrente: Precio fijo especial para clientes frecuentes
- **CA-2.2**: Precio 2 - Descuento por cliente: Descuento porcentual o fijo sobre el precio base
- **CA-2.3**: Precio 3 - Precio por volumen: Precio especial según cantidad comprada

#### 3. Aplicación Automática en Ventas
- **CA-3.1**: Al crear una venta, el sistema debe aplicar automáticamente el mejor precio disponible
- **CA-3.2**: La prioridad de precios debe ser: Precio especial → Precio por volumen → Precio base
- **CA-3.3**: Debe mostrar en el detalle de venta qué precio se aplicó y por qué
- **CA-3.4**: Debe permitir sobreescribir el precio automático si es necesario

#### 4. Validaciones y Reglas de Negocio
- **CA-4.1**: No debe permitir precios especiales mayores al precio base
- **CA-4.2**: Los precios por volumen deben tener rangos válidos (cantidad inicial < cantidad final)
- **CA-4.3**: No debe permitir rangos de volumen superpuestos para el mismo producto/cliente
- **CA-4.4**: Debe validar que el cliente exista y esté activo
- **CA-4.5**: Debe validar que el producto exista y esté activo

#### 5. Reportes y Visibilidad
- **CA-5.1**: Debe existir un endpoint para listar todos los precios especiales configurados
- **CA-5.2**: Debe poder filtrar precios por cliente, producto o tipo de precio
- **CA-5.3**: Debe mostrar el histórico de cambios de precios especiales
- **CA-5.4**: Debe poder exportar la configuración de precios especiales

## Endpoints del Backend

### 1. Gestión de Precios por Cliente

#### POST /api/precios-cliente
**Descripción**: Crear precio especial para un cliente
**Request**:
```json
{
  "empresaId": 1,
  "terceroId": 123,
  "productoPresentacionId": 456,
  "precioEspecial": 85000.00,
  "fechaInicio": "2026-01-01",
  "fechaFin": "2026-12-31",
  "activo": true,
  "observaciones": "Precio especial cliente VIP"
}
```
**Response**: 201 Created

#### GET /api/precios-cliente/{id}
**Descripción**: Obtener precio especial por ID

#### PUT /api/precios-cliente/{id}
**Descripción**: Actualizar precio especial

#### DELETE /api/precios-cliente/{id}
**Descripción**: Eliminar precio especial

#### GET /api/precios-cliente/cliente/{terceroId}
**Descripción**: Listar todos los precios especiales de un cliente

### 2. Gestión de Descuentos por Cliente

#### POST /api/descuentos-cliente
**Descripción**: Crear descuento por cliente
**Request**:
```json
{
  "empresaId": 1,
  "terceroId": 123,
  "categoriaId": 789,
  "porcentajeDescuento": 15.00,
  "tipoDescuento": "porcentaje",
  "fechaInicio": "2026-01-01",
  "fechaFin": "2026-12-31",
  "activo": true
}
```

#### GET /api/descuentos-cliente/{id}
#### PUT /api/descuentos-cliente/{id}
#### DELETE /api/descuentos-cliente/{id}

### 3. Gestión de Precios por Volumen

#### POST /api/precios-volumen
**Descripción**: Crear regla de precio por volumen
**Request**:
```json
{
  "empresaId": 1,
  "productoPresentacionId": 456,
  "rangoVolumenes": [
```

#### PUT /api/precios-volumen/{id}
#### DELETE /api/precios-volumen/{id}

## Validaciones y Reglas de Negocio

### 1. Prioridad de Aplicación de Precios
1. **Primera prioridad**: Precios especiales por cliente (Precio 1)
2. **Segunda prioridad**: Descuentos por cliente (Precio 2)
3. **Tercera prioridad**: Precios por volumen (Precio 3)
4. **Cuarta prioridad**: Precios de lista de precios base

### 2. Validaciones de Negocio
- **VB-001**: Un cliente no puede tener más de un precio especial para el mismo producto
- **VB-002**: Los descuentos por cliente no pueden exceder el 50% del precio base
- **VB-003**: Los rangos de volumen no pueden superponerse entre sí
- **VB-004**: Las fechas de vigencia deben ser coherentes (fechaInicio ≤ fechaFin)
- **VB-005**: Un producto no puede tener más de 5 rangos de volumen configurados
- **VB-006**: Los precios especiales deben estar activos para ser aplicados
- **VB-007**: Los descuentos por cliente solo aplican si el cliente está activo

### 3. Seguridad y Permisos
- **SEG-001**: Solo usuarios con rol `ADMIN` o `GERENTE` pueden crear/actualizar precios
- **SEG-002**: Los usuarios con rol `VENDEDOR` solo pueden ver los precios aplicados
- **SEG-003**: Se debe validar el `empresaId` del JWT en todos los endpoints

### 4. Consistencia de Datos
- **CD-001**: Al eliminar un precio especial, se debe mantener el historial de movimientos
- **CD-002**: Los cambios en precios deben auditarse con usuario y timestamp
- **CD-003**: Los precios por volumen deben mantenerse consistentes con el inventario

## Ejemplos Concretos de Uso

### Ejemplo 1: Cliente Recurrente con Precio Especial
**Cliente**: Juan Pérez (ID: 123)
**Producto**: Laptop Dell (Presentación ID: 456)
**Precio Base**: $1,000,000
**Precio Especial**: $850,000 (15% de descuento)

**Flujo**:
1. Juan Pérez realiza una compra
2. Sistema detecta que es cliente recurrente
3. Aplica precio especial de $850,000
4. Muestra: "Precio especial cliente: $850,000 (15% DTO)"

### Ejemplo 2: Descuento por Categoría
**Cliente**: Distribuidora XYZ (ID: 124)
**Categoría**: Electrónica
**Descuento**: 10% sobre todos los productos de la categoría

**Flujo**:
1. Distribuidora XYZ compra 5 laptops
2. Sistema aplica 10% de descuento sobre el precio base
3. Muestra: "Descuento categoría: 10% DTO - Precio final: $900,000"

### Ejemplo 3: Precios por Volumen
**Producto**: USB 64GB (Presentación ID: 789)
**Rangos de Volumen**:
- 1-9 unidades: $20,000 c/u
- 10-49 unidades: $18,000 c/u
- 50+ unidades: $15,000 c/u

**Flujo**:
1. Cliente compra 25 unidades
2. Sistema aplica rango de 10-49 unidades
3. Precio total: 25 × $18,000 = $450,000
4. Muestra: "Precio por volumen (25 unidades): $18,000 c/u - Total: $450,000"

### Ejemplo 4: Combinación de Reglas
**Cliente**: Distribuidora XYZ (ID: 124)
**Producto**: Laptop Dell (Presentación ID: 456)
**Precio Base**: $1,000,000
**Descuento Cliente**: 10% (por ser cliente mayorista)
**Rango Volumen**: 10+ unidades: $900,000 c/u

**Compra**: 15 unidades

**Aplicación de Reglas**:
1. Primero aplica precio por volumen: $900,000
2. Luego aplica descuento cliente: 10% × $900,000 = $90,000
3. Precio final por unidad: $810,000
4. Total: 15 × $810,000 = $12,150,000

**Muestra**:
- "Precio por volumen: $900,000 c/u (15 unidades)"
- "Descuento cliente: 10% DTO - $90,000"
- "Precio final unidad: $810,000"
- "Total: $12,150,000"

## Integración con Sistema Existente

### 1. Modificaciones a Entidades Existentes
- **ProductoPrecioEntity**: Agregar campo `tipoPrecio` (base, cliente, volumen)
- **ReglaDescuentoEntity**: Agregar campo `aplicaSobre` (base, cliente, volumen)

### 2. Nuevas Entidades Necesarias
```sql
-- Tabla para precios especiales por cliente
CREATE TABLE IF NOT EXISTS precio_cliente (
    id SERIAL PRIMARY KEY,
    empresa_id INT NOT NULL REFERENCES empresa(id),
    tercero_id INT NOT NULL REFERENCES tercero(id),
    producto_presentacion_id INT NOT NULL REFERENCES producto_presentacion(id),
    precio_especial DECIMAL(19,2) NOT NULL,
    fecha_inicio TIMESTAMP,
    fecha_fin TIMESTAMP,
    activo BOOLEAN DEFAULT TRUE,
    observaciones VARCHAR(500),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    deleted_at TIMESTAMP
);

-- Tabla para descuentos por cliente
CREATE TABLE IF NOT EXISTS descuento_cliente (
    id SERIAL PRIMARY KEY,
    empresa_id INT NOT NULL REFERENCES empresa(id),
    tercero_id INT NOT NULL REFERENCES tercero(id),
    categoria_id INT REFERENCES categoria(id),
    porcentaje_descuento DECIMAL(5,2) NOT NULL,
    tipo_descuento VARCHAR(20) DEFAULT 'porcentaje',
    fecha_inicio TIMESTAMP,
    fecha_fin TIMESTAMP,
    activo BOOLEAN DEFAULT TRUE,
    observaciones VARCHAR(500),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    deleted_at TIMESTAMP
);

-- Tabla para precios por volumen
CREATE TABLE IF NOT EXISTS precio_volumen (
    id SERIAL PRIMARY KEY,
    empresa_id INT NOT NULL REFERENCES empresa(id),
    producto_presentacion_id INT NOT NULL REFERENCES producto_presentacion(id),
    cantidad_minima INT,
    cantidad_maxima INT,
    precio_unitario DECIMAL(19,2) NOT NULL,
    activo BOOLEAN DEFAULT TRUE,
    observaciones VARCHAR(500),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    deleted_at TIMESTAMP
);
```

### 3. Modificaciones a Proceso de Venta
- Al calcular el precio de un producto, el sistema debe:
  1. Verificar si existe precio especial para el cliente
  2. Verificar si aplica descuento por cliente
  3. Verificar si aplica precio por volumen según cantidad
  4. Aplicar las reglas en orden de prioridad
  5. Mostrar desglose de precios al vendedor

## Criterios de Aceptación Técnica

### TA-001: Performance
- El cálculo de precios debe completarse en menos de 200ms para 10 items
- Las consultas de precios deben usar índices apropiados

### TA-002: Escalabilidad
- El sistema debe soportar hasta 10,000 reglas de precios por empresa
- Las consultas deben paginarse cuando sea necesario

### TA-003: Seguridad
- Todos los endpoints deben validar el JWT y el `empresaId`
- Los datos sensibles (precios especiales) deben encriptarse en reposo

### TA-004: Auditoría
- Todo cambio en precios debe registrarse con usuario, fecha y motivo
- El sistema debe mantener historial de 2 años de precios

## Pruebas de Aceptación

### Caso de Prueba 1: Precio Especial Cliente
**Dado**: Cliente con precio especial configurado
**Cuando**: Realiza compra del producto
**Entonces**: Se aplica el precio especial y se muestra el desglose

### Caso de Prueba 2: Descuento por Categoría
**Dado**: Cliente con descuento por categoría
**Cuando**: Compra productos de esa categoría
**Entonces**: Se aplica el descuento y se muestra el ahorro

### Caso de Prueba 3: Precio por Volumen
**Dado**: Producto con rangos de volumen
**Cuando**: Compra cantidad que entra en un rango
**Entonces**: Se aplica el precio correspondiente al rango

### Caso de Prueba 4: Combinación de Reglas
**Dado**: Cliente con múltiples reglas aplicables
**Cuando**: Realiza compra
**Entonces**: Se aplican las reglas en orden de prioridad correcto

### Caso de Prueba 5: Validación de Fechas
**Dado**: Regla con fechas de vigencia
**Cuando**: Compra fuera del rango de fechas
**Entonces**: No se aplica la regla y se usa precio base
